# Campaign issues overview

Shareable list of **known issues and limitations** across **whole campaigns** (checkout, summary, bumps, bundles, operators). Written for anyone following a funnel—not only engineers. Technical repro, BS ids, and migration crosswalks stay in the **[template bug log](template-bug-log.md)**.

**Status (short):**


| Label    | Meaning                                                      |
| -------- | ------------------------------------------------------------ |
| Open     | Still a problem or open request; no stable fix yet           |
| Blocked  | Waiting on platform/SDK or a firm product decision           |
| Fixed    | Addressed in templates or process; follow-up QA still useful |
| Verified | Change checked in QA; treat as resolved for that setup       |


---

## Bundle migration note (not a defect)

**Bundle quantity tiers** (`data-next-bundle-selector`, one `packageId` with 1× / 2× / 3× cards) are a **different way to structure a campaign** than the older **multi-package selector** (separate package ids per tier). **Pricing, offers, and rounding** follow the bundle / discount model in the Campaigns app — operators cannot copy the old selector playbook one-for-one.

| 6   | **UX — stacked discount % (early SDK / bundle cards):** When a **cart coupon** (e.g. 10%) applies **on top of** an **already discounted** package (e.g. 50% off retail), **order totals can be correct** while the **single headline savings %** moves to a **combined** figure (e.g. ~**55%**), which **confuses shoppers** who expect **“50% offer” + “extra 10% with code”** as **two** messages. **Desired (SDK):** a **bundle display** slot for **offer-only** % (e.g. **`data-next-bundle-display="savingsOfferPercentage"`**) that **does not** recompute to the all-in % after a coupon; pair with a **separate** coupon badge or line (**`data-next-show="cart.hasCoupon"`** + voucher copy — see **BS-014** in [template bug log](template-bug-log.md)). | **Bundle-tier** checkouts and upsells that show **% off** on tier cards **and** allow **stacked** coupons. | Medium | Open |
| 11  | **Stale package ids / shared origin (empty cart, “Package N not found”):** The SDK resolves packages by **`ref_id`** from the campaign payload — the same numeric **`id`** as **`data-next-package-id`** and **`packageId`** in bundle markup / JSON. **`addItem`** throws **“Package N not found in campaign data”** when **`getPackage(N)`** misses; **N** is not a “missing row” in the JSON you are inspecting. **Confirmed cause — `next-profile-store` (localStorage):** **offer / tier / exit** profiles persist **`activeProfileId`** and **`packageMappings`** (logical slot → **`ref_id`** on the **campaign the profile was built for**). Clear **`next-cart-state`**, **`next-profile-store`**, **`next-campaign-cache_*`** when in doubt; in DevTools **Application**, if package IDs disagree with the API, read **`next-profile-store`**. **Verify:** **`data-next-package-id`** vs live **`ref_id`**. **Markup:** align **`packageId`** to real **`ref_id`** (Olympus placeholder **`1`**). **Diagnostic only:** `<meta name="next-clear-cart" content="true">` — **BS-013**; post-order cleanup — **BS-019**. **Platform (SDK repo):** scope cart, campaign cache, and **profile** by **campaign identity**; **prune** orphan lines / stale profile mappings on campaign load. | **CRO**, **QA**, **exit-intent / profile** flows, shared **preview** / **demo** origins | Medium | Open |
| 10  | Auto-rendered bundle cards (injected from a template) do not bind product name/image the same way as inline cards; placeholders can stay blank. **Low priority:** reference templates use **inline** bundle cards instead, so this SDK path is optional—note for engineering to fix when convenient. | **All** campaigns that opt into Step 4 auto-render (`data-next-bundles` + template id); not used in default olympus-v0.4.              | Low      | Open    |
| 12  | **Cart summary tax row (BS-017):** Three separate gaps mean tax cannot display in the checkout cart summary today: `{tax}` is not passed into the `<template>` vars; `.next-has-tax` is never toggled on `[data-next-cart-summary]`; `data-next-display="cart.tax"` is not implemented. **Workaround:** small custom JS — listen on `cart:updated`, read the tax field from `state.summary`, inject manually, and toggle `.next-has-tax` yourself. | 17  | **BS-020 — `data-next-include-shipping` on bundle selector (no-op):** On **`[data-next-bundle-selector]`**, **`data-next-include-shipping`** is read and stored on the bundle price context but **not passed through** to **`calculateBundlePrice`** (unlike **`PackageSelector`**). **Effect:** setting or omitting the attribute **does not change** tier / bundle headline totals today. **Guidance:** omit on new markup until SDK parity; existing reference attributes are harmless placeholders. **Track:** [template bug log](template-bug-log.md) (**BS-020**). | **All** campaigns using **`data-next-bundle-selector`** with **`data-next-include-shipping`**. | Low | Open |
